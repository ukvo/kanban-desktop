# 📑 ШПАРГАЛКА: РОБОТА З МІГРАЦІЯМИ ALEMBIC (+ UV МЕНЕДЖЕР)

Усі команди виконуються в терміналі WSL2 всередині папки бекенду (`/backend`), де знаходиться файл `pyproject.toml`. Оскільки ми використовуємо `uv`, перед кожною командою додається префікс `uv run`.

---

## 1. Ініціалізація Alembic у проекті (Виконується ОДИН раз)
Створює конфігураційний файл `alembic.ini` та папку `alembic/` зі скриптами міграцій.
```bash
uv run alembic init alembic
```
*Примітка: Для проектів з асинхронною базою даних використовують шаблон `alembic init -t async alembic`, але для нашого SQLite достатньо стандартного варіанту.*

---

## 2. Створення міграцій (Генерація скриптів оновлення)

### Автоматичне створення міграції (Рекомендовано)
Alembic сам порівнює стан моделей у коді (`models.py`) з поточною базою даних і генерує файл міграції.
```bash
uv run alembic revision --autogenerate -m "назва_міграції"
```
*Приклад:* `uv run alembic revision --autogenerate -m "init_db"` або `uv run alembic revision --autogenerate -m "add_task_color_field"`

### Створення порожньої міграції (Для ручного написання SQL-скриптів)
```bash
uv run alembic revision -m "назва_ручної_міграції"
```

---

## 3. Застосування міграцій (Оновлення бази даних)

### Оновлення бази до найновішої версії (Найчастіша команда)
Застосовує всі нові створені міграції до файлу бази даних.
```bash
uv run alembic upgrade head
```

### Оновлення на одну міграцію вперед
```bash
uv run alembic upgrade +1
```

### Перевірка поточної версії бази даних (Яка міграція зараз застосована)
```bash
uv run alembic current
```

### Перегляд історії всіх створених міграцій
```bash
uv run alembic history --verbose
```

---

## 4. Відкат міграцій (Скасування змін)

### Скасування останньої застосованої міграції (Повернення на 1 крок назад)
```bash
uv run alembic downgrade -1
```

### Повне очищення бази (Відкат до самого початку, видалення всіх таблиць)
```bash
uv run alembic downgrade base
```

---

## ⚠️ Важливі правила безпеки при роботі з міграціями:
1. **Завжди перевіряйте згенерований файл:** Навіть при автоматичній генерації (`--autogenerate`), перед застосуванням міграції відкрийте створений файл у папці `alembic/versions/` через VS Code і переконайтеся, що методи `upgrade()` та `downgrade()` містять саме ті зміни, які ви планували.
2. **Ніколи не змінюйте файл міграції, який уже було закоментовано і відправлено на GitHub.** Якщо ви зробили помилку в полі — створіть *нову* міграцію, яка виправляє це поле.
3. **Перед кожним `git pull` від колег (або при зміні гілки):** Бажано виконати `uv run alembic upgrade head`, щоб ваша локальна база даних синхронізувалася з новими файлами міграцій, які прийшли з репозиторію.
