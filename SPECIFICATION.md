# 📋 ТЕХНІЧНЕ ЗАВДАННЯ: ЛОКАЛЬНИЙ ДЕСКТОП KANBAN (Варіант №2)

## 1. Загальний опис та концепція
Створення персонального, локального десктоп-додатка для керування задачами за методологією Kanban. Додаток є повністю автономним, не потребує зовнішнього віддаленого сервера бази даних та працює за принципом **«Offline-First»**. Усі дані користувача зберігаються локально. Синхронізація та архівування виконуються безпосередньо з особистим **Google Диском** користувача за його запитом.

---

## 2. Технологічний стек

### Локальний Бекенд (Папка `/backend`)
* **Мова та фреймворк**: Python 3.12+ / FastAPI (запускається локально на машині користувача).
* **Менеджер пакетів та якість коду**: `uv` + `ruff`.
* **Локальна база даних**: **SQLite** (один файл `kanban.db` у локальній папці додатка).
* **ORM**: SQLModel / SQLAlchemy.
* **Логування**: `loguru` (запис у локальний файл `logs/app.log` з автоматичним zip-стисканням при досягненні 10 МБ, зберігання логів за останні 10 днів).

### Десктоп Фронтенд (Папка `/frontend`)
* **Фреймворк**: Vue 3 (Composition API) + Bun + Vite.
* **Якість коду**: Biome.
* **Стилізація**: SCSS (Світла/Темна теми через CSS-змінні).
* **Контекст Десктопу**: Запуск у вікні браузера через локальний порт (із перспективою пакування в **Tauri** для отримання єдиного `.exe` / `.dmg` файлу).

## 3. Структура даних (SQLite)

### 3.1. Таблиця `projects` (Проекти / Дошки)
* `id`: `INTEGER` (Primary Key, Auto-increment)
* `name`: `VARCHAR(255)` (NOT NULL)
* `description`: `TEXT` (NULL)
* `created_at`: `DATETIME` (DEFAULT CURRENT_TIMESTAMP)

### 3.2. Таблиця `statuses` (Колонки / Статуси задач)
* `id`: `INTEGER` (Primary Key, Auto-increment)
* `name`: `VARCHAR(100)` (NOT NULL)
* `group`: `VARCHAR(20)` (NOT NULL, DEFAULT 'active' -> значення: `active`, `archived`, `deleted`)
* `position`: `INTEGER` (NOT NULL, DEFAULT 0 -> для сортування колонок зліва направо)

### 3.3. Таблиця `tasks` (Картки задач)
* `id`: `INTEGER` (Primary Key, Auto-increment)
* `project_id`: `INTEGER` (Foreign Key -> `projects.id`, ON DELETE CASCADE, Index=True)
* `status_id`: `INTEGER` (Foreign Key -> `statuses.id`, ON DELETE RESTRICT, Index=True)
* `title`: `VARCHAR(255)` (NOT NULL)
* `description`: `TEXT` (NULL, підтримує Markdown)
* `priority`: `INTEGER` (NOT NULL, DEFAULT 2 -> рівні: 1-Low, 2-Medium, 3-High)
* `created_at`: `DATETIME` (DEFAULT CURRENT_TIMESTAMP)
* `updated_at`: `DATETIME` (DEFAULT CURRENT_TIMESTAMP, оновлюється при будь-якій зміні)

### 3.4. Таблиця `subtasks` (Підзадачі / Окремий чек-ліст задач)
* `id`: `INTEGER` (Primary Key, Auto-increment)
* `task_id`: `INTEGER` (Foreign Key -> `tasks.id`, ON DELETE CASCADE, Index=True)
* `text`: `VARCHAR(255)` (NOT NULL)
* `is_done`: `BOOLEAN` (NOT NULL, DEFAULT FALSE)
* `created_at`: `DATETIME` (DEFAULT CURRENT_TIMESTAMP)
* `due_date`: `DATETIME` (NULL -> дедлайн для конкретної підзадачі)

### 3.5. Таблиця `attachments` (Прикріплені файли задач)
* `id`: `VARCHAR(36)` (Primary Key, генерується як UUID для файлового сховища)
* `task_id`: `INTEGER` (Foreign Key -> `tasks.id`, ON DELETE CASCADE, Index=True)
* `original_name`: `VARCHAR(255)` (NOT NULL -> початкова назва файлу, напр. photo.png)
* `local_path`: `VARCHAR(512)` (NOT NULL -> шлях до локальної копії файлу)
* `drive_file_id`: `VARCHAR(255)` (NULL -> ID файлу на Google Диску після успішної синхронізації)
* `uploaded_at`: `DATETIME` (DEFAULT CURRENT_TIMESTAMP)

## 4. Функціональні вимоги (Локальна логіка)

### 4.1. Авторизація та Доступ
* Традиційний логін/пароль відсутній (персональний додаток).
* Авторизація через Google потрібна виключно для доступу до власного Google Диску користувача (через OAuth2), щоб додаток міг читати/писати файли у спеціальну папку додатка на Диску.
* Проект містить опціональну підтримку прапорця в конфігурації `.env` (`APP_MODE=desktop` / `APP_MODE=server`). У разі перемикання додатка в режим `server` (для домашнього хостингу), система вимагатиме ввести єдиний PIN-код/пароль, прописаний користувачем.

### 4.2. Логіка дошки та Картки задачі
* **Drag and Drop**: Нативний HTML5 API (`useDragAndDrop.ts`).
* **Сортування**: Картки в колонці автоматично стають на свої місця на основі `priority` (від більшого до меншого) та дати `updated_at`.
* **Формат відображення**: Коротка картка-віджет на дошці ➡️ Повна форма задачі (модальне вікно) з описом, дедлайнами підзадач та вкладенненнями при кліку.
* **Видалення**: М'яке видалення (перенесення в групу `deleted` — Кошик). Повне видалення (Hard Delete) можливе лише при очищенні Кошика. Видалення колонки (`Status`) заблоковане, якщо всередині є хоча б одна задача.

## 5. Робота з файлами та Синхронізація (Google Диск)

### 5.1. Структура папки на Google Диску користувача
У корені особистого Google Диску додаток створює ізольовану папку `kanban/`:
* `/database_backup/` — архівні копії бази даних (`.zip` файли з міткою часу, наприклад `backup_2026_06_25_1430.zip`).
* `/task_attachments/` — фізичні файли задач (названі за `UUID`).

### 5.2. Механіка Синхронізації та Бекапу (За вимогою)
1. **Поточна робота**: Всі зміни миттєво пишуться в локальний файл бази `kanban.db` та локальну папку для медіа-файлів. Це забезпечує максимальну швидкість без інтернету.
2. **Створення бекапу**: Користувач натискає кнопку **«Синхронізувати з хмарою»**. Local-бекенд бере файл `kanban.db`, стискає його в `.zip` за допомогою вбудованого модуля `zipfile`, додає до назви поточну дату/час і через Google Drive API завантажує у фоновому режимі (`FastAPI BackgroundTasks`) в папку `database_backup/`. Нові прикріплені файли з локальної папки завантажуються в `task_attachments/`.
3. **Обробка колізій (Контроль версій)**: Перед завантаженням бекапу додаток сканує останні мітки `updated_at` задач у хмарі та локально.
   * Якщо версія в хмарі свіжіша за локальну, додаток пропонує оновити локальну базу.
   * Якщо виявлено конфлікт (одна задача змінена на обох пристроях), задача підсвічується червоним (`is_conflicted = true`), і користувач у модальному вікні обирає, яку версію залишити.

# 🗂️ РЕЗЮМЕ ОНОВЛЕНОЇ АРХІТЕКТУРИ (ПЛАН НА ЗАВТРА)

### 1. Зміна концепції Синхронізації (Offline-First)
* Повна відмова від Google Drive API через нестабільність сесій у WSL2 та обмеження доступу.
* Заміна на **Локальний менеджер копій**: ручне створення `.zip` архівів, скачування на ПК, видалення, відновлення бази та імпорт готових архівів через інтерфейс (multipart/form-data) для міграції між хостингом та десктопом.
* Повне винесення менеджера копій на ізольовану сторінку `/backups`.

### 2. Динамічна логіка "Група статусу = Окрема сторінка"
Замість хардкоду окремих екранів "Дошка" та "Кошик", система масштабується за формулою:
`Користувач ➡️ Обирає проект (ID) ➡️ Обирає групу статусів (напр. 'active', 'trash', 'archive')`

* **Ендпоінти роутера фронтенду**: 
  * `/project/:id/group/active` — Основна робоча Kanban-дошка (колоноки Backlog, In Progress, Done).
  * `/project/:id/group/trash` — Сторінка кошика (колонка "Видалені").
  * `/project/:id/group/:any_custom_group` — Будь-яка майбутня кастомна сторінка користувача.
* **Відображення даних**: При переході на таку сторінку додаток відображає лише ті колонки (статуси) проекту, які належать до обраної групи, та автоматично виводить усі задачі, прив'язані до цих колонок.
